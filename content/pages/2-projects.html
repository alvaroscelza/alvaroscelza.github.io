SkollARS

SkollARS es mi emprendimiento personal, cuyo propósito principal es permitir enseñarme a mi mismo. La misión de SkollARS es proporcionar servicios de consultoría y desarrollo de Software a pequeñas y medianas empresas, o particulares que necesiten dichos servicios, pero se mantengan fuera de la audiencia objetivo de grandes Fábricas de Software. La visión de SkollARS es la tecnologización de todas las capas económicas de Uruguay, y eventualmente más allá. Si bien SkollARS como tal, nació en setiembre de 2019, ya desde antes desarrollaba proyectos personales, y en ocasiones a clientes particulares que obtenía informalmente, generalmente familiares y/o amigos.

Mejora de documentación de servicio de control de usuarios

2021 - 1 mes

Nuestro servicio de control de usuarios, cuyo nombre mantengo en privado, contiene un set de reglas que controlan en forma automática las acciones de nuestros usuarios: límites de operación entre otros. El desarrollo y mantenimiento de estas reglas se estaba volviendo insostenible debido a la falta de documentación eficiente al respecto. Nuestro equipo perdió la visión global de todas las reglas aplicadas, los casos en que cada una se ejecutaba, los órdenes de prioridad entre ellas, etc. Se me pidió resolver el problema, ante tal me tomé un tiempo en leer, y analizar el único documento sobre el proyecto: una bitácora de cambios donde se fueron anotando las discusiones conversacionales entre el equipo de desarrollo y el de Compliance. Habiendo forjado en mi mente el historial de cambios realizados pude traducir dicho documento bitácora en una serie de documentos de especificación de requerimientos: diagrama de casos de uso, múltiples diagramas de flujo y un documento arquitectónico de análisis de requerimientos funcionales, requerimientos no funcionales y restricciones del negocio.

Procesos de documentación

2021 - 2 meses

Desde mi ingreso, Ripio ha tenido un crecimiento verdaderamente exponencial en la cantidad de empleados: cerca de un 100% el primer año, y un 87.5% el segundo.
Los procesos de desarrollo y documentación ad hoc utilizados hasta el momento ya no bastaban. Debido a que mi forma de trabajo personal implica un fuerte nivel de aplicación de procesos y metodologías formales, que incluyen diagramación, documentación Kanban, estimación y demás técnicas aprendidas en mi carrera, se me solicitó que utilizara mis conocimientos en pos de mejorar los procesos de Ripio de cara al equipo de desarrollo. Con este fin elaboré un plan de cambios paulatinos procedurales, basado en PDCA.
Este plan comenzaba con documentación técnica formal y estrictamente guiada por apartados en los archivos Readme.md de cada repositorio de código. Luego exigía la inclusión de diagramas arquitectónicos de alto nivel: Allocation View, Components & Connectors View. Y posteriormente, de bajo nivel: Modules View.
As you probably know, this Architectural documentation is based on the book Documenting Software Architectures: Views and Beyond. Finalmente comenzamos a utilizar herramientas de gestión de proyectos, como Jira.

API para pagos con crypto

Python Django PostgreSQL
2020 a 2021 - 1 año

Necesitábamos construir una aplicación para gestionar el nuevo producto de Ripio en Brasil: tarjetas de débito para compras con crypto. Para esto se requería construir una API web con toda la lógica que nuestros equipos de front end consumirían. Esta API tenía dos dificultades principales:
1- Debía consumir los servicios de una API de nuestro proveedor de tarjetas y pagos, Conductor.
2- Debía gestionar los pagos comunicados por nuestro proveedor hacia nuestra BD de balances, con lo cual era necesario un estricto control de dichos pagos, para evitar fallas de seguridad y también de concurrencia, evitando por ejemplo, autorizar y ejecutar pagos dobles.
Fui uno de los dos desarrolladores de esta API.

pre-commit-hooks

Python
2018 a 2019 - 1 año

Pocos meses después de haber entrado a trabajar a BPS y familiarizarme con la herramienta Odoo descubrí que no existían demasiadas herramientas de análisis estático de código para este framework. El mismo sigue un conjunto de buenas prácticas estipuladas por la comunidad, que incluyen, pero no se limitan a pep8.
Esto produce dificultad en verificar el correcto seguimiento de estas buenas prácticas de manera automática.

A fin de intentar subsanar esto me puse a investigar e implementar git hooks para el chequeo automático de nuestro código, al menos respecto a aquellas prácticas más sencillas de implementar con scripts de python. Para esto me basé en el repositorio pre-commit-hooks.

El resultado fue una batería de scripts que se ejecutarían automáticamente cada vez que uno de nuestros desarrolladores hiciera commits de su código, gracias a la librería de python pre-commit.
