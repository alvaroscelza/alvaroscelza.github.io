Mejora de documentación de servicio de control de usuarios

1 mes

Nuestro servicio de control de usuarios, cuyo nombre mantengo en privado, contiene un set de reglas que controlan en forma automática las acciones de nuestros usuarios: límites de operación entre otros. El desarrollo y mantenimiento de estas reglas se estaba volviendo insostenible debido a la falta de documentación eficiente al respecto. Nuestro equipo perdió la visión global de todas las reglas aplicadas, los casos en que cada una se ejecutaba, los ordenes de prioridad entre ellas, etc. Se me pidió resolver el problema, ante tal me tomé un tiempo en leer, y analizar el único documento sobre el proyecto: una bitácora de cambios donde se fueron anotando las discusiones conversacionales entre el equipo de desarrollo y el de Compliance. Habiendo forjado en mi mente el historial de cambios realizados pude traducir dicho documento bitácora en una serie de documentos de especificación de requerimientos: diagrama de casos de uso, múltiples diagramas de flujo y un documento arquitectónico de análisis de requerimientos funcionales, requerimientos no funcionales y restricciones del negocio.

Procesos de documentación

2 meses

Desde mi ingreso, Ripio ha tenido un crecimiento verdaderamente exponencial en la cantidad de empleados: cerca de un 100% el primer año, y un 87.5% el segundo.
Los procesos de desarrollo y documentación ad hoc utilizados hasta el momento ya no bastaban. Debido a que mi forma de trabajo personal implica un fuerte nivel de aplicación de procesos y metodologías formales, que incluyen diagramación, documentación Kanban, estimación y demás técnicas aprendidas en mi carrera, se me solicitó que utilizara mis conocimientos en pos de mejorar los procesos de Ripio de cara al equipo de desarrollo. Con este fin elaboré un plan de cambios paulatinos procedurales, basado en PDCA.
Este plan comenzaba con documentación técnica formal y estrictamente guiada por apartados en los archivos Readme.md de cada repositorio de código. Luego exigía la inclusión de diagramas arquitectónicos de alto nivel: Allocation View, Components & Connectors View. Y posteriormente, de bajo nivel: Modules View.
As you probably know, this Architectural documentation is based on the book Documenting Software Architectures: Views and Beyond. Finalmente comenzamos a utilizar herramientas de gestión de proyectos, como Jira.

API para pagos con crypto

Python Django PostgreSQL
1 año

Necesitábamos construir una aplicación para gestionar el nuevo producto de Ripio en Brasil: tarjetas de débito para compras con crypto. Para esto se requería construir una API web con toda la lógica que nuestros equipos de front end consumirían. Esta API tenía dos dificultades principales:
1- Debía consumir los servicios de una API de nuestro proveedor de tarjetas y pagos, Conductor.
2- Debía gestionar los pagos comunicados por nuestro proveedor hacia nuestra BD de balances, con lo cual era necesario un estricto control de dichos pagos, para evitar fallas de seguridad y también de concurrencia, evitando por ejemplo, autorizar y ejecutar pagos dobles.
Fui uno de los dos desarrolladores de esta API.
